1. Public nonce-minting AJAX endpoint weakens CSRF protection for guest cart actions
Files:
- inc/taiowc.php:78-79
- inc/taiowc.php:719-733
- assets/js/taiowc-cart.js:900-918

Issue:
`taiowc_create_nonces` is exposed through both `wp_ajax_` and `wp_ajax_nopriv_` and returns fresh nonces for `apply-coupon`, `remove-coupon`, and `update-shipping-method` to any unauthenticated caller. In WordPress, nonces are not authentication tokens. Making a public endpoint whose sole purpose is to generate valid nonces for state-changing actions materially reduces the CSRF value of those nonce checks, especially for guest flows.

Suggestion:
Remove the public nonce-generation endpoint. Prefer localizing the required nonce values server-side when the page is rendered, or protect the endpoint with a stronger context check tied to the current session/cart instead of exposing it to arbitrary anonymous requests.

2. Undo-cart AJAX flow uses the wrong nonce and never localizes the nonce it verifies
Files:
- assets/js/taiowc-cart.js:221-232
- inc/taiowc.php:197-211
- inc/taiowc.php:813-816

Issue:
The client sends `taiowc_param.update_qty_nonce` to the `taiowc_undo_item` endpoint, but the server verifies `taiowc_undo_nonce`. At the same time, `wp_localize_script()` never provides `taiowc_undo_nonce` to the frontend. This is a broken nonce implementation in an AJAX handler: legitimate requests cannot satisfy the server-side nonce check, which often leads to disabled functionality or pressure to weaken/remove nonce verification later.

Suggestion:
Create and localize a dedicated `taiowc_undo_nonce`, then submit that value from the undo AJAX request. Alternatively, align both sides on one nonce action name if the intent is to share a single nonce across related cart operations.

3. Tracking-system REST setting is publicly readable
Files:
- inc/cart-tracking/taiowc-cart-track-front.php:256-261

Issue:
The GET route `/taiowc/v1/taiowc-track-system-enabled` uses `permission_callback => '__return_true'`, so any unauthenticated requester can query whether cart tracking is enabled. This is an information disclosure issue. The exposed value is not highly sensitive by itself, but it reveals plugin behavior and expands the public attack surface unnecessarily.

Suggestion:
Restrict the GET route to privileged users such as `manage_options`, or remove the endpoint if the value is only needed in the admin UI. If the value truly must be public, document that decision and keep the response limited to the minimum necessary data.

4. Settings reset nonce is verified from raw request data without normal WordPress unslashing/sanitization
Files:
- inc/taiowc-setting.php:988-999

Issue:
`taiowc_delete_settings()` verifies `$_REQUEST['delete_wpnonce']` directly. This is not an immediate bypass because a nonce check is present, but it is weaker than normal WordPress handling and should be treated as input-validation hardening around a destructive settings action.

Suggestion:
Read the nonce with `isset()`, then pass `sanitize_text_field( wp_unslash( $_REQUEST['delete_wpnonce'] ) )` into `wp_verify_nonce()`. Keep the capability check in place before the delete action.
